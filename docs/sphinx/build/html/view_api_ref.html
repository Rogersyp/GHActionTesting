PTI View API Reference#

Warning

DRAFT DOCUMENTATION - This documentation is currently in draft status and subject to change.

This section provides complete API documentation for the PTI View API (tracing and profiling). The documentation is automatically generated from source code headers.

Note

This page documents the PTI View API for tracing GPU activities. For hardware metrics collection, see PTI Metrics API Reference.

Overview#

The PTI View API provides tracing capabilities for GPU operations. The API includes:

  • Core Functions - Enable/disable tracing, set callbacks, manage views

  • Helper Functions - Utility functions for string conversion

  • Structures - Data structures for profiling records

  • Enumerators - View types, result codes, memory types

  • Function Pointers - Callback function signatures

Core Functions#

  • ptiViewSetCallbacks - Register callback functions

  • ptiViewEnable - Enable a profiling view

  • ptiViewDisable - Disable a profiling view

  • ptiViewGPULocalAvailable - Check local collection support

  • ptiFlushAllViews - Flush all pending records

  • ptiViewGetNextRecord - Retrieve next record

  • ptiViewPushExternalCorrelationId - Push correlation ID

  • ptiViewPopExternalCorrelationId - Pop correlation ID

  • ptiViewGetTimestamp - Get current timestamp

  • ptiViewSetTimestampCallback - Set custom timestamp function

  • ptiViewGetApiIdName - Get API name from ID

  • ptiViewEnableDriverApi - Enable/disable specific driver API

  • ptiViewEnableRuntimeApi - Enable/disable specific runtime API

  • ptiViewEnableDriverApiClass - Enable/disable driver API class

  • ptiViewEnableRuntimeApiClass - Enable/disable runtime API class

Helper Functions#

  • ptiViewOverheadKindToString - Convert overhead kind to string

  • ptiViewMemoryTypeToString - Convert memory type to string

  • ptiViewMemcpyTypeToString - Convert memcpy type to string

Structures#

Record structures contain profiling data passed to callbacks:

  • pti_view_record_base - Base record structure

  • pti_view_record_api - Runtime API call record

  • pti_view_record_kernel - GPU kernel record

  • pti_view_record_memory_copy - Memory copy record

  • pti_view_record_memory_copy_p2p - Peer-to-peer copy record

  • pti_view_record_memory_fill - Memory fill record

  • pti_view_record_synchronization - Synchronization record

  • pti_view_record_external_correlation - External correlation record

  • pti_view_record_overhead - Overhead record

  • pti_view_record_comms - Communication record (oneCCL, Linux only)

Enumerators#

  • pti_result - Function return status codes

  • pti_view_kind - View type identifiers

  • pti_view_external_kind - External correlation types

  • pti_view_overhead_kind - Overhead types

  • pti_view_memory_type - Memory location types

  • pti_view_memcpy_type - Memory copy direction types

  • pti_view_synchronization_type - Synchronization operation types

  • pti_api_group_id - API group identifiers (Level-Zero, SYCL, OpenCL)

  • pti_api_class - API class categories for filtering

Function Pointer Typedefs#

Callback function signatures:

  • pti_fptr_buffer_completed - Buffer completed callback

  • pti_fptr_buffer_requested - Buffer requested callback

Common Usage Patterns#

Filtering Specific APIs:

#include "pti/pti_view.h"
#include "pti/pti_driver_levelzero_api_ids.h"

ptiViewEnable(PTI_VIEW_DRIVER_API);

// Only trace kernel launches and memory copies
ptiViewEnableDriverApi(1, PTI_API_GROUP_LEVELZERO,
                      zeCommandListAppendLaunchKernel_id);
ptiViewEnableDriverApi(1, PTI_API_GROUP_LEVELZERO,
                      zeCommandListAppendMemoryCopy_id);

Decoding API IDs:

void ProcessRecord(pti_view_record_api* rec) {
    const char* api_name = nullptr;
    ptiViewGetApiIdName(rec->_api_group_id, rec->_api_id, &api_name);

    if (api_name) {
        std::cout << "API: " << api_name << std::endl;
    }

    // Or check specific API
    if (rec->_api_group_id == PTI_API_GROUP_LEVELZERO &&
        rec->_api_id == zeCommandListAppendLaunchKernel_id) {
        std::cout << "Found kernel launch!" << std::endl;
    }
}

API ID Naming Convention:

  • Level-Zero: ze<FunctionName>_id

    • zeInit → zeInit_id

    • zeCommandListCreate → zeCommandListCreate_id

  • SYCL/UR: ur<FunctionName>_id

    • urContextCreate → urContextCreate_id

    • urEnqueueKernelLaunch → urEnqueueKernelLaunch_id
